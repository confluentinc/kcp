# Migration Auth Flags Design

## Problem

The `migration execute` command requires a `--credentials-file` flag pointing to a credentials YAML file generated by `kcp discover`. This creates an unnecessary dependency on running discover first. Users migrating a single cluster should be able to provide authentication details directly via flags.

## Solution

Add auth flags to both `migration init` and `migration execute`, following the same pattern used by the old `scan cluster` command. The `--credentials-file` flag is removed from execute. Auth flags + `--source-cluster-arn` together provide everything needed to connect to the source MSK cluster.

The cluster ARN is used to call the MSK API and auto-discover bootstrap brokers. The auth flag determines which set of brokers to use (IAM, SCRAM, TLS, etc.) via `GetBootstrapBrokersForAuthType`.

## Auth Flags

Mutually exclusive boolean flags (choose one):

| Flag | AuthType |
|------|----------|
| `--use-sasl-iam` | `SASL/IAM` |
| `--use-sasl-scram` | `SASL/SCRAM` |
| `--use-tls` | `TLS` |
| `--use-unauthenticated-tls` | `Unauthenticated (TLS Encryption)` |
| `--use-unauthenticated-plaintext` | `Unauthenticated (Plaintext)` |

Conditional credential flags:

| Flag | Required when |
|------|--------------|
| `--sasl-scram-username` | `--use-sasl-scram` |
| `--sasl-scram-password` | `--use-sasl-scram` |
| `--tls-ca-cert` | `--use-tls` |
| `--tls-client-cert` | `--use-tls` |
| `--tls-client-key` | `--use-tls` |

Conditional requirements are enforced in `preRunE` by dynamically marking flags as required (same pattern as old scan cluster command).

## Changes by File

### 1. `cmd/migration/init/cmd_migration_init.go`

- Add auth flag group (mutually exclusive booleans)
- Add SASL/SCRAM credential flag group
- Add TLS credential flag group
- Add `preRunE` conditional requirement logic
- Map bool flags to `config.AuthMode` string
- No other wiring — flags are not used for connecting at init time (future use)
- `--source-cluster-arn` already exists as required

### 2. `cmd/migration/execute/cmd_migration_execute.go`

- Add same auth flag group (duplicated, not shared)
- Add same credential flag groups
- Add `--source-cluster-arn` (required)
- Remove `--credentials-file`
- Add `preRunE` conditional requirement logic
- Map bool flags to `AuthType`, pass credential strings through `MigrationExecutorOpts`

### 3. `cmd/migration/execute/migration_executor.go`

Update `MigrationExecutorOpts`:
- Remove `CredentialsFile string`
- Add `SourceClusterArn string`
- Add `AuthType types.AuthType`
- Add `SaslScramUsername string`
- Add `SaslScramPassword string`
- Add `TlsCaCert string`
- Add `TlsClientCert string`
- Add `TlsClientKey string`

Rewrite `createSourceOffset`:
- Remove credentials file loading (`NewCredentialsFromFile`)
- Remove cluster lookup by ARN in credentials (`FindClusterByArn`)
- Remove auth type extraction from credentials (`GetSelectedAuthType`)
- Use `opts.AuthType` directly
- Build `ClusterAuth` from flag values for `AdminOptionForAuth`
- Keep MSK API bootstrap broker discovery (uses ARN + auth type)

### 4. `internal/types/migration_config.go`

No changes needed. `AuthMode` field already exists.

## No Credential Persistence

Auth credentials (passwords, TLS cert paths) are NOT stored in the migration state file. Users provide auth flags each time they run init or execute. Only the auth type string is stored in `MigrationConfig.AuthMode`.

## Flag Duplication

Auth flags are duplicated between init and execute rather than shared via a common struct/package. The flags are unlikely to change frequently and the duplication keeps each command self-contained.
