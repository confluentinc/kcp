# E2E Migration Test Fixes вҖ” Summary & Context

## Problem

Running `make ci-e2e-tests` fails because the **kcp binary runs on the host machine** but needs to connect to services running inside a Minikube cluster. Three categories of issues were discovered:

1. **Network unreachable** вҖ” K8s-internal DNS names (`*.confluent.svc.cluster.local`) can't be resolved from the host.
2. **Auth mismatch** вҖ” Source Kafka requires SASL/PLAIN authentication, but the test passes `--use-unauthenticated-plaintext`.
3. **TLS failures** вҖ” CFK auto-generated certs use K8s DNS SANs (not `localhost`), and the CA isn't in the host's trust store.
4. **SNI rejection** вҖ” Even after fixing TLS verification, Jetty (REST proxy) rejects connections where the TLS SNI value is `localhost` instead of `destination-restproxy`.

---

## Changes Made

### 1. Port-forwarding (fixes network unreachable)

**Files**: `test/e2e/migration/testdata/setup.sh`, `teardown.sh`

The `.env` file originally contained K8s-internal DNS names:
```
KCP_E2E_SOURCE_BOOTSTRAP=source-kafka.confluent.svc.cluster.local:9071
KCP_E2E_REST_PROXY_ENDPOINT=http://None:8082
```

These aren't resolvable from the host. The REST proxy endpoint was `None` because the service has no ClusterIP (headless service), so the original `kubectl get svc -o jsonpath='{.spec.clusterIP}'` returned `None`.

**Fix**: Added `kubectl port-forward` for three services after all pods are ready:
- Source Kafka вҶ’ `localhost:19071`
- Destination Kafka вҶ’ `localhost:19072`
- REST Proxy вҶ’ `localhost:18082`

PIDs are saved to `.port-forward-pids` so teardown can kill them. The `.env` now uses `localhost:*` endpoints.

The REST proxy endpoint uses `https://` because CFK **requires** TLS on `KafkaRestProxy` resources (removing `tls.autoGeneratedCerts` causes `missing .spec.tls. in spec` error and the pod never starts).

### 2. Source Kafka auth removal (fixes auth mismatch)

**Files**: `test/e2e/migration/testdata/manifests/source-kafka.yaml`, `cluster-link.yaml`

The source Kafka manifest had SASL/PLAIN authentication on its internal listener, but the E2E test passes `--use-unauthenticated-plaintext`. kcp doesn't have a `--use-sasl-plain` flag for the source cluster.

**Fix**: Removed the `authentication` block from the source Kafka internal listener and the corresponding auth block from the cluster link's `sourceKafkaCluster` spec. The source Kafka now accepts unauthenticated plaintext connections, matching what the test expects.

### 3. `--insecure-skip-tls-verify` flag (fixes TLS verification)

**Files**:
- `internal/client/kafka_admin.go` вҖ” Added `insecureSkipVerify` field to `AdminConfig`, `WithInsecureSkipVerify()` option function, and threaded the flag through all TLS config helper functions (`configureSASLTypeOAuth`, `configureSASLTypeSCRAM`, `configureSASLTypePlain`, `configureUnauthenticated`).
- `cmd/migration/init/cmd_migration_init.go` вҖ” Added `--insecure-skip-tls-verify` flag (optional, defaults to `false`).
- `cmd/migration/init/migration_initializer.go` вҖ” When flag is set, creates HTTP client with `InsecureSkipVerify` TLS config.
- `cmd/migration/execute/cmd_migration_execute.go` вҖ” Same flag.
- `cmd/migration/execute/migration_executor.go` вҖ” When flag is set, creates insecure HTTP client AND passes `WithInsecureSkipVerify()` to both source and destination Kafka client creation.
- `internal/client/kafka_admin_test.go` вҖ” Updated test call sites to match new function signatures (added `bool` parameter).

**Why code changes were needed**: Two reasons `SSL_CERT_FILE` (the standard Go env var for custom CA certs) doesn't work here:
1. On macOS, Go uses the Security framework (via cgo) for cert verification and **ignores `SSL_CERT_FILE`** even with `CGO_ENABLED=0`. Confirmed by testing: `SSL_CERT_FILE=/path/to/ca.crt go run test.go` fails with "certificate is not trusted".
2. Even if the CA were trusted, CFK auto-generated certs have SANs like `destination-restproxy`, `destination-restproxy.confluent.svc.cluster.local` вҖ” **not `localhost`**. Hostname verification would still fail.

This flag follows the standard CLI pattern (`kubectl --insecure-skip-tls-verify`, `helm --tls-skip-verify`).

### 4. SNI-free TLS dialer (partially fixes SNI rejection)

**Files**: `cmd/migration/init/migration_initializer.go`, `cmd/migration/execute/migration_executor.go`

When `--insecure-skip-tls-verify` is set, the HTTP client uses a custom `DialTLSContext` that creates a TLS connection **without setting `ServerName`**. This means Go's TLS client sends no SNI extension in the ClientHello. The intent was that Jetty would fall back to its default certificate when no SNI is provided (Jetty only rejects *invalid* SNI, not *missing* SNI when `sniRequired` is false).

**Current status**: This does NOT fully work. Jetty still returns "Invalid SNI" even with empty SNI. The Confluent REST Proxy's Jetty configuration appears to require a valid SNI regardless.

---

## Remaining Issue: SNI

The REST proxy (Jetty-based) rejects connections when the TLS SNI doesn't match `destination-restproxy`. This is a **server-side check** вҖ” the client cannot bypass it.

### Possible solutions (not yet implemented)

| Approach | Pros | Cons |
|----------|------|------|
| **A. /etc/hosts entry** вҖ” Add `127.0.0.1 destination-restproxy` to `/etc/hosts`, use `https://destination-restproxy:18082` as endpoint | Simple, no code changes | Requires `sudo`, not CI-friendly |
| **B. CFK configOverrides** вҖ” Disable Jetty SNI check via REST proxy config property | No code changes, no sudo | Unknown property name; CP REST Proxy may not expose this setting |
| **C. Custom certs with `localhost` SAN** вҖ” Generate TLS certs including `localhost` in SANs, use CFK `secretRef` instead of `autoGeneratedCerts` | Fixes SNI at the source | Complex cert generation in setup.sh; still need `SSL_CERT_FILE` to work (it doesn't on macOS) |
| **D. Run kcp inside a pod** вҖ” Build for linux, `kubectl cp` binary into a pod, run via `kubectl exec` | All DNS/TLS/SNI works natively | Significant test refactor; need kubeconfig inside pod; cross-compilation |
| **E. Send correct SNI from client** вҖ” Set `TLSClientConfig.ServerName` to match the REST proxy's expected hostname | Targeted fix | Need to know the expected hostname; could add `--cluster-rest-server-name` flag or derive from service name |

**Recommended next step**: Option **A** (add `/etc/hosts` in setup.sh with sudo) is the simplest. Option **E** (add a `--cluster-rest-server-name` flag) is cleaner long-term. Option **D** is the most robust but highest effort.
