# E2E Migration Test — SNI Fix & In-Pod Execution

## Problem

After the fixes in [e2e-test-fixes.md](e2e-test-fixes.md), the E2E tests still failed with the remaining SNI issue:

```
HTTP ERROR 400 Invalid SNI
URI: https://localhost:18082/kafka/v3/clusters/.../links/e2e-link/mirrors
```

The REST proxy (Jetty 12) rejects TLS connections where the SNI doesn't match a SAN in its certificate. The kcp binary ran on the host machine, connecting via `kubectl port-forward` to `localhost:18082`. Go's TLS client sent `SNI=localhost`, but the auto-generated certificate only covered `destination-restproxy` and K8s DNS names.

---

## Approaches Tried

### 1. Custom DialTLSContext with empty SNI (from previous session)

**Status**: Already existed, didn't work.

The previous session added a custom `DialTLSContext` in `migration_initializer.go` and `migration_executor.go` that created a TLS connection without setting `ServerName`, sending an empty SNI extension. Jetty still rejected it — it validates SNI before checking other TLS settings and rejects both invalid *and* missing SNI.

### 2. `--cluster-rest-server-name` CLI flag

**Status**: Rejected — scope creep.

Proposed adding a new `--cluster-rest-server-name` flag to `migration init` and `migration execute` that would set `tls.Config.ServerName` to send the correct SNI (`destination-restproxy`) while connecting to `localhost`. This would have worked technically, but was rejected because adding CLI flags solely to make tests pass defeats the purpose of the E2E tests. The test should validate the tool as users would use it.

### 3. Custom TLS certificate with `localhost` SAN

**Status**: Broke REST proxy → Kafka connectivity.

Generated a server certificate with `localhost` in the SANs (signed by the existing CA), stored it as a K8s secret (`destination-restproxy-tls`), and changed the REST proxy manifest from `autoGeneratedCerts: true` to `secretRef: destination-restproxy-tls`.

The TLS handshake with the client succeeded — Jetty accepted `SNI=localhost` because the cert included it. However, the REST proxy returned **404 for all endpoints**, including `/kafka/v3/clusters`. Root cause: switching from `autoGeneratedCerts` to `secretRef` broke CFK's internal trust chain. With `autoGeneratedCerts`, CFK automatically configures both the keystore (server cert) and truststore (CA certs for Kafka connectivity). With `secretRef`, only the keystore was configured — the REST proxy could no longer validate destination-kafka's TLS certificate, causing all Kafka API calls to fail.

### 4. `/etc/hosts` entry

**Status**: Rejected — not CI-friendly.

Proposed adding `127.0.0.1 destination-restproxy` to `/etc/hosts` in `setup.sh`, changing the endpoint to `https://destination-restproxy:18082`, and reverting to `autoGeneratedCerts`. Go would send `SNI=destination-restproxy` (from URL hostname), Jetty would accept it, and CFK's trust chain would remain intact.

This worked but was rejected because:
- Requires `sudo` on the CI machine
- Semaphore CI (`s1-prod-ubuntu24-04-amd64-1`) may not have sudo access
- Mutates global system state on shared CI runners
- Cleanup in teardown is error-prone (crashes leave stale entries)

### 5. Run kcp inside a pod (adopted solution)

**Status**: Implemented.

Instead of fighting networking from the host, run the kcp binary inside the Minikube cluster where K8s DNS resolves service names natively and TLS SANs match.

---

## Solution: In-Pod Execution

### Architecture

```
Host Machine                          Minikube (kcp-e2e)
┌─────────────────┐                  ┌──────────────────────────────────────┐
│  E2E Test        │                  │  confluent namespace                  │
│  (go test)       │                  │                                       │
│                  │   kubectl exec   │  kcp-runner pod (alpine)              │
│  kubectl exec ───┼─────────────────┼──► /workspace/kcp binary              │
│                  │                  │     ├── source-kafka:9071 (plaintext) │
│  kubectl get ────┼── assertions ───┼──► destination-kafka:9071 (TLS)       │
│  (Gateway CR,    │   (host-side    │     └── destination-restproxy:8082    │
│   pod UIDs)      │    kubeconfig)  │          (HTTPS, auto-gen certs)      │
│                  │                  │          SNI = destination-restproxy ✓│
│                  │   kubectl exec   │                                       │
│  cat state.json ─┼─────────────────┼──► /workspace/migration-state.json    │
└─────────────────┘                  └──────────────────────────────────────┘
                                       No port-forwarding needed
                                       K8s DNS resolves service names
                                       TLS SANs match natively
```

### How it works

1. `setup.sh` cross-compiles kcp for Linux (`GOOS=linux go build`)
2. A runner pod (`alpine:3.20`) is deployed with a ServiceAccount that has RBAC permissions for Gateway CRs and pods
3. The binary and fixture YAMLs are copied into the pod via `kubectl cp`
4. A kubeconfig is generated from the SA token inside the pod (kcp requires `--kube-path`)
5. The Go test runs kcp via `kubectl exec` instead of direct execution
6. Endpoints use K8s-internal DNS names — no port-forwarding, no SNI issues
7. The test reads the migration state file from the pod via `kubectl exec cat`
8. Assertions (Gateway CR, pod rollout) still use the host kubeconfig via K8s client-go

### Why this eliminates all networking issues

| Issue | Why it's gone |
|-------|---------------|
| **SNI mismatch** | URL hostname matches cert SANs (K8s DNS names) |
| **Port-forwarding** | Not needed — pod reaches services directly |
| **TLS trust chain** | CFK auto-generated certs work natively in-cluster |
| **DNS resolution** | K8s DNS resolves `source-kafka`, `destination-restproxy`, etc. |
| **Host dependencies** | No `/etc/hosts`, no `sudo`, no custom certs |

### Code simplification

The broken `DialTLSContext` workaround from the previous session was removed from both `migration_initializer.go` and `migration_executor.go`. It was replaced with a standard `http.Transport{TLSClientConfig: &tls.Config{InsecureSkipVerify: true}}`. This is not a behavioral change — the `--insecure-skip-tls-verify` flag works identically. The custom dialer was unnecessary complexity that didn't solve the SNI problem.
